# Plan 05-02: Contextual Action Options for Reach-Out Screen

**Phase:** 5 - Reach-Out Intelligence
**Wave:** 2 (depends on 05-01 being merged — carousel in place)
**Requirements:** REACH-02

## Goal

Replace the static action button list in the reach-out flow with contextually ranked action options that vary between people based on their interaction history, relationship type, and available contact info. Action labels and ordering should never be identical for every person.

## Tasks

### Task 1: Create reachOutActionEngine

**File:** `src/lib/reachOutActionEngine.ts` (NEW)

**Actions:**
1. Define a `ReachOutAction` interface:
   ```typescript
   interface ReachOutAction {
     type: InteractionType;
     label: string;
     sublabel?: string; // e.g. "Your usual way to connect"
     isPrimary: boolean;
   }
   ```
2. Create `getReachOutActions(person: Person, interactions: Interaction[]): ReachOutAction[]` function:
   a. Count interaction types for this person from their history
   b. Determine the person's most-used channel (most frequent interaction type)
   c. Build an ordered action list:
      - If person has interaction history: put their most-used channel first as primary, with a contextual sublabel
      - If no history: use relationship-type defaults (e.g. family -> "Call" first; friend -> "Message" first; colleague -> "Message" first; mentor -> "Message" first)
      - Always include "Meet in person" (available for everyone)
      - Filter: call/video only if `person.phone` exists
   d. Generate varied labels per action based on context:
      - Message: "Send a message" / "Drop them a note" / "Say hello" (varies by relationship)
      - Call: "Give them a call" / "Call and catch up" / "Ring them"
      - Video: "Video chat" / "Face-to-face call"
      - In-person: "Meet up" / "Get together" / "Plan something together"
   e. Add sublabels for the primary action:
      - "Your favorite way to connect" (if most-used)
      - "A warm way to reach out" (if default by relationship)
3. Export the function and the `ReachOutAction` type.

**Verification:** Function returns different orderings for different people; person with phone gets call/video options; person without phone does not; labels vary based on relationship type and history.

### Task 2: Integrate contextual actions into BridgeScreen

**File:** `app/reach-out/[id].tsx` (MODIFY)

**Actions:**
1. Import `getReachOutActions` from `@/lib/reachOutActionEngine`
2. Import `usePersonInteractions` from `@/hooks`
3. In `BridgeScreen`, call `usePersonInteractions(person.id)` to get interaction history
4. Call `getReachOutActions(person, interactions)` to get the ordered action list
5. Replace the hardcoded action buttons section with a data-driven render:
   - Map over the action list from the engine
   - First action (isPrimary=true) gets the sage-green primary button styling
   - Remaining actions get the white secondary button styling
   - Each button shows the icon (mapped from type: message->MessageCircle, call->Phone, video->Video, in_person->Users)
   - Each button shows the label from the engine
   - If the action has a sublabel, show it below the label in smaller warmGray text
6. Remove the hardcoded "Suggested opening" chips section — the contextual actions replace the static suggestions
7. Update the primary text above actions to vary:
   - If person has recent memories: keep existing contextual copy
   - If person has interaction history: "Stay connected with {name}"
   - Default: "Sometimes the simplest reach-out matters most"

**Verification:** Opening reach-out for Sarah (friend, has phone, frequent messager) shows different action order than Dr. Patel (mentor, no phone). Action labels are not identical between people. Sublabel appears on primary action. No static "Suggested opening" chips.
